## Summary

Phase 5.5 adds the revenue layer to AXIOM: deal commission invoicing (2% sale / 1.5% rental) and agency subscription billing (Starter 500 EGP/Pro 1200 EGP/Enterprise 2500 EGP), payable via Paymob (card/wallet iframe) or Fawry (offline cash reference codes).

The backend work is a new `app/payments/` module that follows the exact same thin-router / service-functions / Pydantic schemas pattern used by every existing module in this codebase. Three new DB tables (`payments`, `deals`, `agency_subscriptions`) plus an ALTER to the `agencies` table are needed. Both gateways use HMAC-based webhook verification — Paymob uses SHA-512, Fawry uses SHA-256. Idempotency is enforced via a UNIQUE constraint on `merchant_ref` in the `payments` table.

The frontend adds a DealReportModal, PaymentMethodSelector, PaymobCheckout (iframe embed), FawryCheckout (reference code display + 30-second polling), AgencySubscriptionCard, and a payments tab in the dashboard.

**Primary recommendation:** Follow the existing router/schemas pattern precisely. Keep all monetary values in piasters (EGP × 100) in the DB; convert to EGP only at the API response layer. Use `httpx.AsyncClient` for all outbound gateway calls (no synchronous blocking in async routes).

### Pattern 3: Paymob 3-Step Legacy API Flow

Paymob Egypt uses a 3-step "Accept" API. Each step must complete synchronously before the next:

```python
# paymob.py
BASE_URL = "https://accept.paymob.com/api"

async def get_auth_token(api_key: str) -> str:
    """Step 1: Exchange API key for short-lived auth token."""
    async with httpx.AsyncClient() as client:
        resp = await client.post(
            f"{BASE_URL}/auth/tokens",
            json={"api_key": api_key},
        )
        resp.raise_for_status()
        return resp.json()["token"]

async def register_order(auth_token: str, amount_cents: int, merchant_order_id: str) -> str:
    """Step 2: Register order with Paymob. Returns order_id."""
    async with httpx.AsyncClient() as client:
        resp = await client.post(
            f"{BASE_URL}/ecommerce/orders",
            json={
                "auth_token": auth_token,
                "delivery_needed": False,
                "amount_cents": amount_cents,
                "currency": "EGP",
                "merchant_order_id": merchant_order_id,
                "items": [],
            },
        )
        resp.raise_for_status()
        return str(resp.json()["id"])

async def get_payment_key(
    auth_token: str,
    order_id: str,
    amount_cents: int,
    integration_id: int,
    billing_data: dict,
) -> str:
    """Step 3: Obtain payment_key for iframe embedding."""
    async with httpx.AsyncClient() as client:
        resp = await client.post(
            f"{BASE_URL}/acceptance/payment_keys",
            json={
                "auth_token": auth_token,
                "amount_cents": amount_cents,
                "expiration": 3600,
                "order_id": order_id,
                "billing_data": billing_data,
                "currency": "EGP",
                "integration_id": integration_id,
                "lock_order_when_paid": True,
            },
        )
        resp.raise_for_status()
        return resp.json()["token"]

# Iframe URL:
# https://accept.paymob.com/api/acceptance/iframes/{PAYMOB_IFRAME_ID}?payment_token={payment_key}
```

### Pattern 4: Fawry Charge Creation

Fawry uses a single POST call. The PAYATFAWRY method generates a cash reference code valid at Fawry POS outlets.

```python
# fawry.py
import hashlib

STAGING_URL = "https://atfawry.fawrystaging.com/ECommerceWeb/Fawry/payments/charge"
PRODUCTION_URL = "https://atfawry.com/ECommerceWeb/Fawry/payments/charge"

def _fawry_charge_signature(
    merchant_code: str,
    merchant_ref: str,
    customer_profile_id: str,
    payment_method: str,
    amount: str,  # "0.00" format
    secure_key: str,
) -> str:
    """SHA-256 of concatenated fields (no separator)."""
    data = merchant_code + merchant_ref + customer_profile_id + payment_method + amount + secure_key
    return hashlib.sha256(data.encode()).hexdigest()

async def create_fawry_charge(
    merchant_ref: str,
    customer_mobile: str,
    amount_egp: float,
    description: str,
) -> dict:
    """Returns {'referenceNumber': '963455678', 'statusCode': 200}."""
    amount_str = f"{amount_egp:.2f}"
    sig = _fawry_charge_signature(
        settings.fawry_merchant_code,
        merchant_ref,
        customer_mobile,  # used as customerProfileId
        "PayAtFawry",
        amount_str,
        settings.fawry_secure_key,
    )
    async with httpx.AsyncClient() as client:
        resp = await client.post(
            STAGING_URL if settings.fawry_sandbox else PRODUCTION_URL,
            json={
                "merchantCode": settings.fawry_merchant_code,
                "merchantRefNum": merchant_ref,
                "customerProfileId": customer_mobile,
                "customerMobile": customer_mobile,
                "paymentMethod": "PayAtFawry",
                "currencyCode": "EGP",
                "amount": amount_str,
                "signature": sig,
                "chargeItems": [{"itemId": merchant_ref, "description": description, "price": amount_str, "quantity": 1}],
            },
        )
        resp.raise_for_status()
        return resp.json()
```

### Pattern 5: Paymob HMAC Webhook Verification

Paymob uses SHA-512. The fields must be concatenated in this exact lexicographic order:

```python
# paymob.py
import hashlib, hmac

PAYMOB_HMAC_FIELDS = [
    "amount_cents",
    "created_at",
    "currency",
    "error_occured",
    "has_parent_transaction",
    "id",
    "integration_id",
    "is_3d_secure",
    "is_auth",
    "is_capture",
    "is_refunded",
    "is_standalone_payment",
    "is_voided",
    "order.id",
    "owner",
    "pending",
    "source_data.pan",
    "source_data.sub_type",
    "source_data.type",
    "success",
]

def verify_paymob_hmac(transaction_obj: dict, received_hmac: str, hmac_secret: str) -> bool:
    """Verify Paymob webhook HMAC-SHA512."""
    def get_nested(obj: dict, key: str) -> str:
        parts = key.split(".")
        val = obj
        for part in parts:
            val = val.get(part) if isinstance(val, dict) else None
        return str(val) if val is not None else ""

    concatenated = "".join(get_nested(transaction_obj, f) for f in PAYMOB_HMAC_FIELDS)
    calculated = hmac.new(
        hmac_secret.encode(),
        concatenated.encode(),
        hashlib.sha512,
    ).hexdigest()
    return hmac.compare_digest(calculated, received_hmac)
```

### Pattern 6: Fawry Callback Signature Verification

Fawry sends HTTP POST to your webhook URL. Uses SHA-256. Field concatenation order:

```python
# fawry.py

def verify_fawry_callback(params: dict, secure_key: str) -> bool:
    """Verify FawryPay server notification v2 messageSignature."""
    # Concatenation order per FawryPay docs:
    # requestId + fawryRefNumber + merchantRefNumber + paymentAmount +
    # orderAmount + fawryFees + orderStatus + paymentMethod + secureKey
    data = (
        str(params.get("requestId", ""))
        + str(params.get("fawryRefNumber", ""))
        + str(params.get("merchantRefNumber", ""))
        + str(params.get("paymentAmount", ""))
        + str(params.get("orderAmount", ""))
        + str(params.get("fawryFees", ""))
        + str(params.get("orderStatus", ""))
        + str(params.get("paymentMethod", ""))
        + secure_key
    )
    calculated = hashlib.sha256(data.encode()).hexdigest()
    return hmac.compare_digest(calculated, params.get("messageSignature", ""))
```

### Pattern 7: Idempotent Webhook Handler

Duplicate webhook deliveries (network retry) must not double-process:

```python
# router.py
@router.post("/paymob/callback", status_code=200)
async def paymob_callback(request: Request):
    body = await request.json()
    obj = body.get("obj", {})
    received_hmac = body.get("hmac", "")

    if not verify_paymob_hmac(obj, received_hmac, settings.paymob_hmac_secret):
        raise HTTPException(status_code=400, detail="Invalid HMAC")

    merchant_ref = str(obj.get("order", {}).get("merchant_order_id"))
    success = obj.get("success", False)

    if not success:
        return {"status": "ignored"}  # failed transaction

    # Idempotency: check if already processed
    existing = (
        supabase_admin.table("payments")
        .select("id, status")
        .eq("merchant_ref", merchant_ref)
        .single()
        .execute()
    )
    if existing.data and existing.data["status"] == "paid":
        return {"status": "already_processed"}  # 200, no duplicate

    # Mark paid and cascade to deal/subscription
    await payment_service.mark_payment_paid(merchant_ref, provider="paymob")
    return {"status": "ok"}
```

### Anti-Patterns to Avoid

- **Synchronous httpx in async route:** Never use `httpx.get()`/`httpx.post()` directly — always `async with httpx.AsyncClient()`. Blocking calls stall the event loop.
- **Logging with print():** All gateway call failures must use `logging.getLogger(__name__).error(...)`.
- **Storing EGP as float in DB:** Floating-point rounding causes commission errors. Store piasters (integer) in DB, convert at response layer.
- **Trusting callback without HMAC:** The webhook endpoints have no JWT auth — HMAC is the only protection. Never skip it.
- **Passing raw Request body through Pydantic for webhooks:** Webhook bodies may be nested (`obj.success`). Parse with `request.json()` not Pydantic body injection — Pydantic validation will break on Paymob's nested structure.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Constant-time comparison | `==` string compare | `hmac.compare_digest()` | String `==` is vulnerable to timing attacks |
| SHA hash generation | Custom hash function | `hashlib.sha256` / `hashlib.sha512` | stdlib, battle-tested |
| Async HTTP | `requests` or `urllib` | `httpx.AsyncClient` | Already in project; non-blocking |
| Form validation | Manual field checking | React Hook Form + Zod | Already in project; consistent |
| Polling (Fawry) | `setInterval` in useEffect | TanStack Query `refetchInterval` | Handles cleanup, visibility pausing, error backoff |

**Key insight:** Both gateway signature schemes are SHA-based hash-of-concatenation — no custom crypto needed. The stdlib `hmac` and `hashlib` modules handle everything.

## Common Pitfalls

### Pitfall 1: Piaster/EGP Confusion in Commission

**What goes wrong:** Commission is computed as `deal_value * 0.02`, producing EGP floats. These get stored directly in the DB, then multiplied by 100 somewhere else, ending up 100x inflated.
**Why it happens:** The `deal_value` in the API request body is in EGP (e.g., `2500000`). The DB stores `commission_amount` in piasters. If you forget to multiply by 100 before INSERT, payments are off.
**How to avoid:** Define a clear boundary — API in/out is always EGP; DB always stores piasters. Write a single `to_piasters(egp: float) -> int` and `to_egp(piasters: int) -> float` helper in `service.py`. Use these exclusively.
**Warning signs:** Commission amounts in the dashboard look like `500.00` instead of `50,000`.

### Pitfall 2: Paymob Auth Token Expiry Mid-Flow

**What goes wrong:** You call Step 1 (get auth token), then something slow happens (DB query, another HTTP call), then Step 3 fails with 401.
**Why it happens:** Paymob auth tokens are short-lived (~1 hour but the exact TTL is not documented). Network latency on slow connections compounds this.
**How to avoid:** Execute all 3 Paymob steps in a single `async def` without yielding control between them (no `await` between steps other than the gateway calls themselves). Do not cache the auth token across requests.
**Warning signs:** Intermittent 401 errors from Paymob Step 3 in production but not in tests.

### Pitfall 3: Fawry Amount Format

**What goes wrong:** Fawry requires `"580.55"` — a string with exactly two decimal places. Passing `580.5` or `580` fails signature validation because the `amount` in the signature must exactly match the body.
**Why it happens:** Python `str(float)` drops trailing zeros: `str(580.50) == "580.5"`.
**How to avoid:** Always format as `f"{amount:.2f}"` before both signature generation and the request body. These two must be bit-for-bit identical.
**Warning signs:** Fawry returns status code indicating signature mismatch even though the key looks correct.

### Pitfall 4: Webhook HMAC on Paymob — Nested Fields

**What goes wrong:** `source_data.pan`, `source_data.sub_type`, `source_data.type`, and `order.id` require dot-notation access into a nested JSON object. Naively treating the flat query-string fields will compute a wrong HMAC.
**Why it happens:** Paymob sends the callback as a JSON body with a nested `obj` containing the transaction, and a top-level `hmac` field. Some implementations incorrectly try to verify against query params.
**How to avoid:** Parse `await request.json()`, extract `body["obj"]` as the transaction object, and use the nested accessor shown in Pattern 5.
**Warning signs:** Every Paymob callback returns 400 (HMAC invalid) in production but your test passes.

### Pitfall 5: Deal Reported Twice

**What goes wrong:** A user submits the DealReportModal twice (double-click, network retry). Two deal rows are created for the same listing, doubling the commission.
**Why it happens:** No uniqueness constraint on `(listing_id, owner_id)` in the deals table.
**How to avoid:** Add a UNIQUE constraint on `deals(listing_id, owner_id)`. In the service, catch the unique violation and return a 409 with a helpful message.
**Warning signs:** Users complain about seeing duplicate deals in their history.

### Pitfall 6: Missing conftest.py patch for payments.router

**What goes wrong:** The existing `conftest.py` `mock_supabase` fixture patches 14 specific modules. The new `app.payments.router` and any sub-modules (`app.payments.paymob`, `app.payments.fawry`) that call `supabase_admin` directly need to be added to that patch list.
**Why it happens:** Python `unittest.mock.patch` patches the name in the specific module where it is imported, not the original definition.
**How to avoid:** Add all new module patches to the existing `mock_supabase` fixture in `tests/conftest.py`. Also patch `app.payments.paymob` and `app.payments.fawry` for outbound httpx calls using `patch("httpx.AsyncClient")`.
**Warning signs:** Tests fail with `supabase_admin` raising unexpected calls, or integration tests try to actually call Paymob/Fawry.

## DB Schema Decisions

### New Table: `payments`

```sql
CREATE TYPE payment_status   AS ENUM ('pending', 'paid', 'failed', 'refunded');
CREATE TYPE payment_provider AS ENUM ('paymob', 'fawry');
CREATE TYPE payment_type     AS ENUM ('deal_commission', 'agency_subscription');

CREATE TABLE payments (
  id             uuid           PRIMARY KEY DEFAULT gen_random_uuid(),
  user_id        uuid           NOT NULL REFERENCES profiles(id) ON DELETE CASCADE,
  payment_type   payment_type   NOT NULL,
  amount         integer        NOT NULL,  -- piasters (EGP * 100)
  currency       text           NOT NULL DEFAULT 'EGP',
  provider       payment_provider NOT NULL,
  payment_method text,                    -- 'card', 'wallet', 'fawry'
  status         payment_status NOT NULL DEFAULT 'pending',
  merchant_ref   uuid           UNIQUE NOT NULL DEFAULT gen_random_uuid(),  -- idempotency key
  provider_ref   text,          -- Paymob order_id or Fawry referenceNumber
  fawry_ref      text,          -- 10-digit cash reference code
  deal_id        uuid           REFERENCES deals(id) ON DELETE SET NULL,
  subscription_id uuid          REFERENCES agency_subscriptions(id) ON DELETE SET NULL,
  created_at     timestamptz    DEFAULT now(),
  paid_at        timestamptz
);

CREATE INDEX idx_payments_user     ON payments (user_id);
CREATE INDEX idx_payments_merchant ON payments (merchant_ref);

ALTER TABLE payments ENABLE ROW LEVEL SECURITY;
CREATE POLICY payments_owner_read ON payments FOR SELECT USING (user_id = auth.uid());
-- Writes only via service role key (FastAPI backend)
```

**Key decisions:**
- `merchant_ref` is a UUID generated by our system (not by the gateway). It is the idempotency key. UNIQUE constraint prevents duplicate webhook processing.
- `amount` is in piasters (integer). No floating-point rounding risk.
- `provider_ref` stores the gateway's own transaction ID for refunds/disputes.

### ALTER TABLE: `agencies`

```sql
-- Add subscription state denormalized onto agencies for fast reads
-- (canonical source remains agency_subscriptions table)
ALTER TABLE agencies
  ADD COLUMN subscription_plan       subscription_plan NOT NULL DEFAULT 'none',
  ADD COLUMN subscription_expires_at timestamptz,
```

## Code Examples

### Deal Commission Computation

```python
# Source: derived from ROADMAP.md requirements + Python stdlib
# service.py

import os
from decimal import Decimal, ROUND_HALF_UP

COMMISSION_RATE_SALE    = float(os.getenv("COMMISSION_RATE_SALE", "0.02"))
COMMISSION_RATE_RENTAL  = float(os.getenv("COMMISSION_RATE_RENTAL", "0.015"))

def compute_commission(deal_type: str, deal_value_egp: float) -> tuple[float, int]:
    """
    Returns (commission_pct_display, commission_amount_piasters).
    deal_value_egp: EGP amount from API request body.
    """
    rate = COMMISSION_RATE_SALE if deal_type == "sale" else COMMISSION_RATE_RENTAL
    commission_egp = Decimal(str(deal_value_egp)) * Decimal(str(rate))
    commission_piasters = int(commission_egp.quantize(Decimal("1"), rounding=ROUND_HALF_UP) * 100)
    return rate * 100, commission_piasters  # (2.0, 5000000) for a 2.5M EGP sale
```

```typescript
      api.get<ApiFawryStatus>(`/api/payments/fawry/status/${merchantRef}`),
    refetchInterval: (data: ApiFawryStatus | undefined) =>
      data?.status === "pending" ? 30_000 : false,  // stop when no longer pending
    enabled,
  }),
};
```

## Open Questions

1. **Paymob billing_data required fields**
   - What we know: Step 3 payment key request requires a `billing_data` object with customer information
   - What's unclear: Exact required vs optional fields (first_name, last_name, email, phone, street, building, floor, apartment, city, country, state, postal_code all appear in documentation examples but it's unclear which are truly required)
   - Recommendation: Set all fields to "NA" for unknown values as Paymob's example code suggests for B2B flows

2. **Paymob test-mode credentials**
   - What we know: Paymob has a sandbox/test environment; the test card number from their docs is `4987654321098769`
   - What's unclear: Whether the PAYMOB_API_KEY for test mode is different from production, and whether the IFRAME_ID changes between environments
   - Recommendation: Plan 55-01 should include a note to configure test-mode credentials before implementation

3. **Fawry sandbox vs production base URL**
   - What we know: Staging is `atfawry.fawrystaging.com`; production is `atfawry.com`
   - What's unclear: Whether `FAWRY_SANDBOX=true` is enough or a separate merchant_code is needed per environment
   - Recommendation: Use a `FAWRY_SANDBOX` boolean that switches the base URL; merchant accounts for staging vs prod are typically separate

4. **agency_subscriptions quota enforcement**
   - What we know: `listing_quota` is stored on `agencies` table; the `PLAN_CONFIG` dict in `agencies/router.py` already defines quotas (starter=10, pro=50, enterprise=9999)
   - What's unclear: Whether quota enforcement (blocking new listing creation when over quota) should be implemented in Phase 5.5 or is out of scope
   - Recommendation: Implement basic quota check in `POST /api/listings` — read `agencies.listing_quota` vs `agencies.listings_used`; raise 402 if exceeded
